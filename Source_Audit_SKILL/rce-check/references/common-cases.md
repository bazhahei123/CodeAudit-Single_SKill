# Command Execution Source Common Cases

## Purpose

This file defines shared command execution and code execution source concepts, source classification logic, propagation patterns, false-positive controls, and evidence standards that apply across languages and frameworks.

Use this file as the base reference for command execution source review before loading any language-specific reference.

This file explains:
- what an execution source point is,
- how to distinguish source, propagation, execution boundary, context, sink, and control from a source-review perspective,
- how to classify trust boundaries,
- how to reason about shell, argv, eval, interpreter, and external tool sources,
- when to record `Confirmed source`, `Suspected source`, `Not enough evidence`, or `Probably irrelevant`.

This reference is guidance, not proof. Do not report a vulnerability only because code contains a source described here. Always verify the real source origin, real propagation path, and real downstream execution relevance in the target code.

---

# 1. Core Concepts

## 1.1 Execution source point

An execution source point is where data that may later influence command execution, shell interpretation, interpreter invocation, expression evaluation, script execution, or external tool behavior enters or becomes available to the backend code path.

Examples:
- `host` query parameter used by diagnostics
- uploaded filename passed to a converter
- selected tool name
- command option or mode
- shell command template
- eval expression
- queue job payload
- stored automation task
- environment variable value
- working directory value

A source point is an audit starting point, not proof of a vulnerability.

## 1.2 Source, propagation, execution boundary, sink, and context

### Source
The origin of data that may be copied, formatted, joined, templated, stored, or passed toward an execution boundary.

Examples:
- `host` from query string
- `tool` from request body
- `args` from queue payload
- `expression` from admin form
- `task.command_template` from database
- uploaded file path used by a CLI converter

### Propagation
The way a source is copied, normalized, escaped, quoted, split, concatenated, formatted, templated, stored, or transformed before reaching an execution sink.

Examples:
- shell command string interpolation
- argv list assembly
- option prefixing
- script file generation
- environment map construction
- stored task later loaded by worker

### Execution boundary
The point where application data begins influencing execution semantics.

Examples:
- selecting an executable
- adding argv entries
- building a shell string
- setting environment variables
- passing stdin to an interpreter
- evaluating an expression

### Sink
The API or helper that actually executes code, commands, tools, expressions, or scripts.

Examples:
- OS command APIs
- shell wrappers
- process builders
- eval/exec APIs
- expression engines
- script engines
- external tool wrappers

### Execution context
The context that determines how the source is interpreted.

Examples:
- shell context
- argv context
- eval/expression context
- interpreter context
- external tool option context
- environment/context-variable context

Source discovery should document enough downstream use to explain why the source matters.

# 2. Trust Boundary Classification

## 2.1 Client-controlled
The value comes from request path, query string, request body, form data, headers, cookies, GraphQL variables, RPC arguments, uploaded filenames, uploaded metadata, or user-supplied import rows.

## 2.2 External-system-controlled
The value comes from a queue message, webhook, partner payload, provider event, internal service response, job payload, or integration sync. It is only trusted after origin, integrity, and scope are verified.

## 2.3 Stored attacker-influenced
The value is read from database, cache, object storage, job storage, saved task definitions, command templates, admin/replay storage, configuration records, or automation metadata, and earlier attacker influence is possible.

## 2.4 Server-trusted
The value is fixed in trusted server-side code, selected from a strict allowlist, generated by a trusted scheduler, or derived from trusted configuration with no attacker-controlled content.

## 2.5 Mixed
The value combines untrusted input with trusted data, or an untrusted value is mapped, allowlisted, constrained, escaped, or transformed before use.

## 2.6 Unclear
The origin cannot be determined from visible code.

# 3. Shared Source Categories

## 3.1 Command and tool selection sources
Values that select what executable, script, tool, subcommand, plugin, or interpreter is used.

Examples:
- command name
- tool name
- script name
- executable path
- subcommand
- action mode

## 3.2 Shell string and template sources
Values that become part of shell strings or command templates.

Examples:
- shell fragment
- command template
- script snippet
- formatted command string
- pipeline fragment
- redirect target

## 3.3 Argument and option sources
Values that become argv entries, flags, options, modes, file paths, URLs, or target values for tools.

Examples:
- host argument
- filename argument
- output path
- `--option` value
- mode selector
- config file path

## 3.4 Eval and expression sources
Values interpreted by eval-like APIs, script engines, expression engines, or rule engines.

Examples:
- expression string
- formula
- script body
- rule definition
- dynamic code fragment

## 3.5 Environment, working-directory, and input stream sources
Values that influence the execution environment rather than the command line itself.

Examples:
- environment variable
- working directory
- stdin
- config path
- temporary script file content

## 3.6 Stored and second-order execution sources
Values written earlier and executed later.

Examples:
- saved command template
- stored automation task
- queued job argument
- scheduled script definition
- replay payload
- admin task configuration

# 4. Shell, argv, eval, and tool source contexts

## 4.1 Shell context sources
Sources that enter `sh -c`, `bash -c`, `cmd /c`, PowerShell, shell scripts, or string-form shell execution.

Audit relevance:
Shell context sources are high priority because shell parsing can interpret separators, substitutions, expansions, redirections, and quoting.

## 4.2 Argv context sources
Sources that enter process APIs as argument arrays.

Audit relevance:
Argv context is usually safer than shell context, but attacker-controlled command names, options, file paths, and modes can still be execution-relevant.

## 4.3 Eval/expression context sources
Sources that enter eval, exec, expression engines, script engines, or dynamic rule engines.

Audit relevance:
The source may be interpreted as code or privileged expression syntax.

## 4.4 External tool context sources
Sources that influence CLIs, converters, scanners, media tools, archive tools, OCR/AI wrappers, network utilities, or automation scripts.

Audit relevance:
Tool-specific option parsing and side effects matter even without shell interpretation.

# 5. Shared Source Patterns

## S1. Request value becomes command argument
Example idea:
- host, file, URL, mode, or target value from request is passed into a command wrapper.

Audit relevance:
The source should be traced to determine whether it is shell, argv, option, or tool-specific context.

## S2. Request value selects command or tool
Example idea:
- `tool`, `action`, `script`, or `command` comes from request input.

Audit relevance:
Dynamic command selection is a high-priority source even when argv arrays are used.

## S3. Stored command template
Example idea:
- database task template, saved automation rule, or admin config later builds a command.

Audit relevance:
Stored values can create second-order execution source paths.

## S4. Eval or expression body
Example idea:
- formula, rule, script, template, or expression text is stored or submitted and later evaluated.

Audit relevance:
Source classification should distinguish fixed safe expressions from attacker-controlled code-like input.

## S5. Option-like input
Example idea:
- user value can start with `-`, becomes a flag, or controls tool mode.

Audit relevance:
Argument-array execution can still be dangerous when options are attacker-controlled.

## S6. File path or environment source
Example idea:
- uploaded file path, output path, config path, environment variable, or working directory influences a tool.

Audit relevance:
These sources may alter tool behavior, file access, plugin loading, or script execution.

# 6. False-Positive Controls

Do not record a source point as high-priority if:
- the value is fixed in trusted code,
- the command, subcommand, options, and expression are all selected from strict allowlists,
- the value is used only for display, logging, or labels,
- the downstream operation does not use the value in an execution-relevant context,
- the source cannot be influenced by an attacker or weakly trusted producer.

Use `Suspected source` or `Not enough evidence` if:
- the source is visible but downstream execution behavior is hidden,
- the sink is visible but source origin is hidden,
- a wrapper helper may enforce allowlists but its implementation is unavailable,
- quoting, escaping, or shell behavior is abstracted away,
- stored task writer paths are missing.

Do not over-claim based only on:
- helper names like `run`, `exec`, `invoke`, or `tool`,
- process API presence,
- `shell=False` by itself,
- quoting helpers,
- an admin-only label,
- a stored command feature without visible attacker influence.

# 7. Source Classification

## Confirmed source
Use `Confirmed source` when there is clear evidence that:
- the source origin is visible,
- it reaches command construction, argument construction, tool selection, eval/expression input, or an execution wrapper,
- and the trust boundary can be classified.

## Suspected source
Use `Suspected source` when:
- a value appears execution-relevant,
- the origin or downstream use is partially visible,
- but hidden wrapper, allowlist, quoting, parser, scheduler, or storage behavior may change the classification.

## Not enough evidence
Use `Not enough evidence` when:
- the source origin cannot be determined,
- the downstream execution use cannot be determined,
- or critical wrapper/execution behavior is not visible.

## Probably irrelevant
Use `Probably irrelevant` when:
- the value is visible,
- downstream use is visible,
- and it does not influence command/tool/script selection, argv/options, shell strings, eval/expression content, environment, working directory, or stored execution state.

# 8. What Good Evidence Looks Like

Strong source point records usually include:
- the exact entry point
- the source value name
- the source origin and trust boundary
- propagation steps such as formatting, joining, quoting, argv assembly, storage, or wrapper calls
- the first downstream execution-relevant use
- context if visible: shell, argv, eval, interpreter, external tool, environment, or worker
- the follow-up sink/control check needed

Good source points usually answer:
1. Where does the execution-relevant value enter?
2. Who controls or can influence it?
3. Is it command selection, shell text, argv, option, eval content, environment, file path, or stored task data?
4. What execution behavior should be checked next?
5. What code proves the connection?

---

# 9. Shared Follow-up Guidance

After source discovery, the command execution audit should verify:
- whether the source reaches a real execution sink,
- whether command names and tools are fixed or strictly allowlisted,
- whether argv construction avoids shell interpretation where appropriate,
- whether option-like input is constrained,
- whether shell strings or templates include untrusted content,
- whether eval/expression engines run only safe fixed subsets,
- whether stored and queued execution data is attacker-influenced,
- whether wrapper helpers enforce narrow execution contracts.

Avoid weak conclusions such as:
- source exists, therefore vulnerability exists,
- `shell=False` exists, therefore all input is safe,
- quoting exists, therefore shell input is safe,
- wrapper exists, therefore execution is constrained,
- admin path exists, therefore source is trusted.

---

# 10. Shared Quick Checklist

Use this as a reminder, not as a substitute for reasoning.

- Is there a client-controlled or weakly trusted execution-relevant source?
- Does it influence command name, tool name, subcommand, script, expression, or interpreter?
- Does it become shell text, argv, option, file path, environment, cwd, stdin, or config?
- Is it stored or queued before later execution?
- Is the execution context shell, argv, eval, interpreter, wrapper, or external tool?
- What sink and control should review this source next?
