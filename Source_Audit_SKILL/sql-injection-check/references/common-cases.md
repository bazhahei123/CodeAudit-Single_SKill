# SQL Injection Source Common Cases

## Purpose

This file defines shared SQL injection source concepts, source classification logic, structural SQL source types, propagation patterns, false-positive controls, and evidence standards that apply across languages and frameworks.

Use this file as the base reference for SQL injection source review before loading any language-specific reference.

This file explains:
- what a SQL-relevant source point is,
- how to distinguish source, propagation, query boundary, execution behavior, and control from a source-review perspective,
- how to classify trust boundaries,
- how to reason about value sources, structural sources, raw fragment sources, ORM helper sources, and second-order sources,
- when to record `Confirmed source`, `Suspected source`, `Not enough evidence`, or `Probably irrelevant`.

This reference is guidance, not proof. Do not report a vulnerability only because code contains a source described here. Always verify the real source origin, real propagation path, and real downstream SQL relevance in the target code.

---

# 1. Core Concepts

## 1.1 SQL-relevant source point

A SQL-relevant source point is where data that may later influence SQL query values, SQL structure, query builder behavior, raw SQL fragments, report queries, stored procedure behavior, or data-access helper behavior enters or becomes available to the backend code path.

Examples:
- `keyword` query parameter used by search
- `sort` or `direction` query parameter used by list endpoints
- `filter[field]` and `filter[operator]` from a request body
- uploaded import row value used in a report query
- saved filter condition loaded from a database
- report template selected by an admin UI
- queue payload that includes query options
- stored dashboard metadata that selects columns
- ORM raw-helper argument passed through a repository wrapper

A source point is an audit starting point, not proof of a vulnerability.

## 1.2 Source, propagation, query boundary, and control

### Source
The origin of data that may be copied, normalized, mapped, formatted, joined, stored, or passed toward SQL-relevant logic.

Examples:
- `q` from query string
- `sortBy` from request body
- `operator` from GraphQL variables
- `whereClause` from a saved filter record
- `report.columns` from database
- `table` from admin configuration
- `limit` from an API request

### Propagation
The way a source is copied, normalized, escaped, cast, mapped, allowlisted, concatenated, formatted, templated, stored, or transformed before reaching SQL-relevant logic.

Examples:
- filter DTO to repository method
- request sort field to query builder option
- raw condition stored and later read by a worker
- report template expanded into query text
- user option mapped to a fixed server-side column name

### Query boundary
The point where application data begins influencing SQL semantics.

Examples:
- adding WHERE criteria
- selecting a column or table
- selecting ORDER BY field or direction
- setting LIMIT or OFFSET
- passing raw SQL text to a helper
- expanding a saved report template
- passing dynamic values to a stored procedure that builds SQL internally

### Control
The protection or constraint that determines whether a source remains only a safe value or can affect SQL structure.

Examples:
- real bound parameters for values
- strict allowlists for structural elements
- fixed server-side mappings
- safe typed query builder APIs
- stored filter schemas that represent allowed operations instead of raw SQL
- revalidation before stored values are reused

Source discovery should document enough downstream use to explain why the source matters.

# 2. Trust Boundary Classification

## 2.1 Client-controlled
The value comes from request path, query string, request body, form data, headers, cookies, GraphQL variables, RPC arguments, uploaded metadata, or user-supplied import rows.

## 2.2 External-system-controlled
The value comes from a queue message, webhook, partner payload, provider event, internal service response, job payload, or integration sync. It is only trusted after origin, integrity, and scope are verified.

## 2.3 Stored attacker-influenced
The value is read from database, cache, object storage, job storage, saved filters, report templates, dashboard definitions, admin query options, stored query metadata, or imported records, and earlier attacker influence is possible.

## 2.4 Server-trusted
The value is fixed in trusted server-side code, selected from a strict allowlist, generated by trusted backend logic, or derived from trusted configuration with no attacker-controlled content.

## 2.5 Mixed
The value combines untrusted input with trusted data, or an untrusted value is mapped, allowlisted, constrained, cast, bound, or transformed before use.

## 2.6 Unclear
The origin cannot be determined from visible code.

# 3. Shared Source Categories

## 3.1 Value source points
Values intended to become SQL data values.

Examples:
- username
- email
- status
- keyword
- numeric ID
- date range
- tenant ID

Audit relevance:
These sources usually require real binding or safe ORM value APIs downstream.

## 3.2 Search and filter source points
Values that define dynamic query criteria.

Examples:
- search term
- filter key
- filter operator
- filter value
- advanced search condition
- report parameter

Audit relevance:
Filter keys and operators may become structural sources, while filter values may be safe only if bound.

## 3.3 Sort and pagination source points
Values that control result ordering and result size.

Examples:
- sort field
- sort direction
- limit
- offset
- page size
- cursor

Audit relevance:
Sort fields and directions are structural SQL sources. Numeric pagination values still need origin and constraint checks.

## 3.4 Structural SQL source points
Values that select SQL structure rather than only data values.

Examples:
- table name
- column name
- select field list
- operator
- raw WHERE fragment
- JOIN target
- GROUP BY field
- SQL function name
- schema or partition name

Audit relevance:
Prepared statement placeholders generally do not protect these. Expect strict allowlists or fixed mappings.

## 3.5 Raw fragment and template source points
Values that become part of executable SQL text or SQL-like templates.

Examples:
- raw where clause
- order clause
- native query template
- report SQL template
- saved filter expression
- stored procedure dynamic SQL text

Audit relevance:
Raw fragments are high-priority source points when they can be influenced by a weakly trusted producer.

## 3.6 Stored and second-order SQL source points
Values written earlier and reused later in SQL-relevant logic.

Examples:
- saved search
- saved filter
- report template
- dashboard column selection
- admin-configured condition
- queued query job
- imported metadata later used in queries

Audit relevance:
Stored values require writer-path review and revalidation before query use.

# 4. SQL Source Contexts

## 4.1 Bound value context
Sources that are passed as values to a safe parameter binding API.

Audit relevance:
Usually lower risk, but verify true binding, correct placeholder use, and no mixed unsafe structural fragments.

## 4.2 Query string context
Sources that are concatenated, interpolated, formatted, or templated into SQL text.

Audit relevance:
High priority because the source may alter SQL structure, logic, comments, quoting, or expressions.

## 4.3 Structural context
Sources that select columns, tables, operators, functions, sort fields, direction, limit, offset, or raw clauses.

Audit relevance:
High priority because ordinary value placeholders usually do not apply.

## 4.4 ORM and query builder context
Sources passed into ORM or query builder APIs.

Audit relevance:
Safe APIs may bind values correctly, but raw helpers, text fragments, dynamic field names, and bypass helpers can remain SQL-relevant.

## 4.5 Stored or generated query context
Sources that are stored, generated, expanded, or executed later.

Audit relevance:
The immediate writer path and the later reader path must both be understood.

# 5. Shared Source Patterns

## S1. Request value becomes query value
Example idea:
- username, keyword, status, or ID from request is passed toward a repository or query helper.

Audit relevance:
Trace whether it is bound safely, inserted into query text, or mixed with structural fragments.

## S2. Request value selects sort or filter structure
Example idea:
- `sort`, `direction`, `field`, or `operator` comes from request input.

Audit relevance:
Structural sources should be mapped to fixed server-side choices.

## S3. Raw WHERE or ORDER fragment source
Example idea:
- request, admin input, or saved filter supplies a raw clause or expression.

Audit relevance:
Raw fragments can carry direct SQL logic and need strict constraints or redesign.

## S4. Dynamic table, column, or select-list source
Example idea:
- report code accepts selected columns, table names, or schema names from request or stored configuration.

Audit relevance:
These are structural sources and cannot be made safe by value binding alone.

## S5. ORM raw helper input source
Example idea:
- a repository wrapper accepts a string and passes it to a raw query API, native query API, or raw builder fragment.

Audit relevance:
The wrapper may hide the query boundary, so source evidence should include the wrapper call and the argument origin.

## S6. Saved filter or report template source
Example idea:
- saved report criteria, dashboard metadata, or admin-defined filter is later expanded into a query.

Audit relevance:
This creates a second-order source path; identify who can write the stored value and whether it is revalidated.

# 6. False-Positive Controls

Do not record a source point as high-priority if:
- the value is fixed in trusted code,
- the source is only used for display, logging, or labels,
- the downstream operation does not use the value in SQL-relevant logic,
- the structural value is selected from a strict server-side allowlist or fixed mapping,
- the source cannot be influenced by an attacker or weakly trusted producer.

Use `Suspected source` or `Not enough evidence` if:
- the source is visible but downstream query behavior is hidden,
- the query boundary is visible but source origin is hidden,
- a wrapper helper may enforce binding or allowlists but its implementation is unavailable,
- ORM/query builder behavior is abstracted away,
- stored filter writer paths are missing.

Do not over-claim based only on:
- parameter names like `query`, `filter`, `sort`, or `where`,
- presence of raw helper names without visible source flow,
- presence of `prepare(...)` alone,
- ORM or repository usage alone,
- a dynamic field that is clearly allowlisted.

# 7. Source Classification

## Confirmed source
Use `Confirmed source` when there is clear evidence that:
- the source origin is visible,
- it reaches query construction, query option construction, raw fragment creation, ORM raw helper input, report-query generation, or stored SQL metadata,
- and the trust boundary can be classified.

## Suspected source
Use `Suspected source` when:
- a value appears SQL-relevant,
- the origin or downstream use is partially visible,
- but hidden wrapper, allowlist, binding, mapper, storage, or query-builder behavior may change the classification.

## Not enough evidence
Use `Not enough evidence` when:
- the source origin cannot be determined,
- the downstream SQL relevance cannot be determined,
- or critical wrapper/query behavior is not visible.

## Probably irrelevant
Use `Probably irrelevant` when:
- the value is visible,
- downstream use is visible,
- and it does not influence SQL values, SQL structure, raw fragments, query builder options, stored query metadata, or data-access behavior.

# 8. What Good Evidence Looks Like

Strong source point records usually include:
- the exact entry point
- the source value name
- the source origin and trust boundary
- propagation steps such as DTO mapping, query option assembly, filtering, casting, allowlist mapping, storage, or wrapper calls
- the first downstream SQL-relevant use
- context if visible: value binding, query string, structural option, ORM helper, raw fragment, stored template, or worker
- the follow-up query/control check needed

Good source points usually answer:
1. Where does the SQL-relevant value enter?
2. Who controls or can influence it?
3. Is it a value, filter key, operator, sort field, table, column, raw fragment, query template, ORM helper input, or stored query setting?
4. What downstream SQL behavior should be checked next?
5. What code proves the connection?

---

# 9. Shared Follow-up Guidance

After source discovery, the SQL injection audit should verify:
- whether the source reaches real query construction or execution,
- whether values use real parameter binding,
- whether structural elements are fixed or strictly allowlisted,
- whether ORM/query builder APIs are safe for the specific argument position,
- whether raw fragments include weakly trusted content,
- whether stored filters and report templates are attacker-influenced,
- whether equivalent routes, exports, reports, and admin paths enforce the same constraints.

Avoid weak conclusions such as:
- source exists, therefore vulnerability exists,
- ORM exists, therefore all SQL input is safe,
- `prepare(...)` exists, therefore structural input is safe,
- escaping exists, therefore query text is safe,
- admin path exists, therefore source is trusted.

---

# 10. Shared Quick Checklist

Use this as a reminder, not as a substitute for reasoning.

- Is there a client-controlled or weakly trusted SQL-relevant source?
- Does it influence a value, filter, sort, operator, column, table, raw fragment, template, or stored query setting?
- Does it become query text, query builder input, ORM raw input, report criteria, or stored query metadata?
- Is it stored or queued before later query use?
- Is the context value binding, structural selection, raw fragment, ORM helper, stored procedure, report template, or worker?
- What query/control check should review this source next?
